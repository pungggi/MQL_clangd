Introducing MQL Debugging Pre-Release

We are thrilled to introduce a highly requested feature: Integrated Debugging for MQL!

Why Pre-Release? This is released early to gather your feedback. It is fully functional but undergoing active testing. If it proves valuable and stable, we'll integrate it permanently. Try it out and let us know what you think!

How it Works

Unlike traditional debuggers, this solution uses a custom Debug Bridge. When you start debugging, the extension automatically instruments your MQL code with lightweight diagnostic macros, compiles it, and launches it in MetaTrader. It then communicates with VS Code via file I/O to simulate a real debugging session.

Step-by-Step Guide

  • Set Breakpoints
    Open any .mq4, .mq5, or .mqh file and set breakpoints by clicking in the editor's left gutter.
  • Start Debugging
    Click the Start Debugging (bug icon) button in the editor's top title bar, or press Ctrl+Alt+D. The extension will instrument and compile your code into a special debug build (named *.mql_dbg_build).
  • MetaTrader Launches & Auto-Attaches the EA
    MetaTrader opens automatically. If your source file lives inside the terminal's MQL5\Experts\ folder, the debug EA is attached to the first open chart automatically — no manual drag-and-drop required. Simply switch to MetaTrader and let it run.

    If your workspace is outside the Experts folder, MetaTrader will still open but you will need to find the *.mql_dbg_build EA in the Navigator panel and drag it onto a chart manually.
  • Inspect the State
    When a breakpoint is hit, VS Code will pause. Explore your variables in the Run and Debug panel and view your Call Stack.
  • Resume Execution
    Use the VS Code debug toolbar to click Continue (or press F5) to let your code run until the next breakpoint. Note: Step Over, Step Into, and Step Out all behave like Continue — they resume execution to the next breakpoint rather than stepping one line at a time (see Known Limitations below).
  • Stop Debugging
    When you are finished, click Stop in VS Code. The extension will automatically send a stop command to MetaTrader, causing the debug EA/Script to self-unload from the chart.

What Variables are Tracked?

To keep performance high, the debugger smartly tracks variables based on your settings:

  • Locals & Parameters: Tracks local variables and function parameters accessed near your breakpoints.
  • Class Members: Monitors fields belonging to the current class instance (this).
  • Deep Analysis (Optional): Want more details? Enable mql_tools.Debug.DetailLevel = "deepAnalysis" in your VS Code settings. This will additionally track global primitives and recursively expand object fields.
  • Breakpoint Info: Provides metadata such as hit counts, source file names, line numbers, and timestamps.

Advanced Features

  • Conditional Breakpoints: Right-click a breakpoint and add a condition — telemetry only fires when the condition is true.
  • Hit Conditional Breakpoints: Right-click a breakpoint, choose Edit Hit Count…, and enter a hit-count expression. The breakpoint only pauses when the condition is met.
    ExpressionMeaningExample use
    5 or = 5 Pause on the 5th hit exactly Skip the first 4 ticks and inspect only the 5th
    > 10 Pause after more than 10 hits Let an EA warm up, then start inspecting
    % 100 Pause every 100th hit Sample a hot loop without pausing every iteration
    Other supported operators: >=, <, <=.
  • Logpoints: Right-click the gutter, choose Add Logpoint…, and type a message template. The EA prints to the VS Code Debug Console without pausing — ideal for tracing live execution.

    Use {expression} placeholders to interpolate variable values at runtime:
    Example: For a logpoint on a line where spread and Ask are in scope, enter:
    Tick: spread = {spread}, Ask = {Ask}

    Output in the Debug Console:
    Tick: spread = 12, Ask = 1.08542
    Logpoints can also be combined with hit conditions — for example, set a hit count of % 50 together with a logpoint message to log every 50th tick instead of every tick.
  • Manual // @watch annotations: The debugger auto-detects local variables near each breakpoint, but some values — like global variables, computed properties, or array elements — may be out of reach. Add a // @watch comment within 5 lines above or 2 lines below the breakpoint to explicitly name them: 
    double gSpread;          // global — not auto-detected

void OnTick()
{
   double ask = SymbolInfoDouble(_Symbol, SYMBOL_ASK);
   double bid = SymbolInfoDouble(_Symbol, SYMBOL_BID);
   gSpread = ask - bid;

   // @watch gSpread ask bid        <— names on one line
   int ticket = OrderSend(…);   // <— breakpoint set here
}
    The debugger will now capture gSpread, ask, and bid alongside any auto-detected locals when this breakpoint is hit.
  • Call Stack Tracking: Functions containing breakpoints automatically report ENTER/EXIT events so the dashboard shows a live call stack.
⚠ Pause / Continue: Each breakpoint injects a blocking spin-wait. The EA thread is fully paused — no OnTick/OnTimer events fire. Use only on demo accounts or in the Strategy Tester. The EA auto-resumes after 120 seconds as a safety failsafe.

Known Limitations

  • No Step Over / Step Into / Step Out: Because the debugger communicates with MetaTrader via file-based I/O (not a live process attachment), it cannot control execution at the instruction level. All stepping actions behave like Continue — they resume to the next breakpoint. To inspect intermediate state, set additional breakpoints on the lines you want to observe.
  • No Expression Evaluation: The Debug Console does not support evaluating arbitrary MQL expressions at runtime. Use // @watch annotations or enable Deep Analysis to capture the variables you need.
